# MCPMate AI模块：文本MCP配置提取器

## 1. 背景信息

### 1.1 功能规划
**当前阶段**：文本内容 → MCP服务配置JSON（最小化原型）
**未来规划**：AI推荐最佳config suite + 服务器 + 工具组合

### 1.2 技术架构决策
- **推理库**：Candle-rs（Hugging Face的Rust推理库）
- **模型格式**：GGUF格式千文模型（0.5B参数）
- **模型路径**：`/Users/Loocor/.mcpmate/models/qwen2.5-0.5b-instruct-q4.gguf`
- **存储策略**：模型存储在用户目录，不在代码库中
- **缓存策略**：不做缓存，每次重新推理（避免幻觉问题）
- **性能预期**：模型加载约2秒，单次推理约1-2秒
- **技术原则**：坚持基于真实LLM推理，绝不使用规则匹配

### 1.3 开发模式
- **独立开发**：独立crate，暂不与主程序耦合
- **输出方式**：控制台打印JSON，不保存文件
- **集成计划**：后期整合到主API端点

## 2. 最小化原型需求

### 2.1 核心流程（4步）
1. **模型加载**：从指定路径加载千文GGUF模型
2. **提示词初始化**：设定MCP配置提取的系统提示词
3. **文本输入与推理**：读取测试文本，执行模型推理
4. **结果输出**：将推理结果解析为JSON格式并打印

### 2.2 功能边界
**当前阶段只做：**
- 单文件实现（如果可能）
- 控制台应用程序
- 硬编码模型路径
- 简单的测试文本输入
- 基础错误处理

**当前阶段不做：**
- Web API接口
- 复杂的配置管理
- 多模型支持
- 缓存机制
- 数据库集成
- 用户界面

### 2.3 成功标准
- 能成功加载指定的千文模型
- 能基于系统提示词进行文本推理
- 能输出符合MCP配置格式的JSON
- 整个流程能在控制台稳定运行

## 3. 渐进式技术策略

### 3.1 技术选型原则
- **坚持LLM推理**：绝不使用规则匹配或模式匹配
- **渐进式开发**：从最简单的功能开始，逐步扩展
- **单一职责**：每个组件只做一件事
- **错误透明**：遇到问题时与用户讨论，不自行降级

### 3.2 核心依赖
- **Candle-rs**: Rust原生LLM推理库
- **Tokenizers**: 分词器支持
- **Serde**: JSON序列化
- **Anyhow**: 错误处理

### 4.3 测试用例
**输入示例**：
“一个简单的文件服务器，可以读取和写入本地文件系统中的文件”

**期望输出格式（正确的MCP配置结构）**：
```json
{
  "mcpServers": {
    "file-server": {
      "command": "python",
      "args": [
        "-m",
        "file_server"
      ]
    }
  }
}
```

**配置格式说明**：
- 顶层为 `mcpServers` 对象
- 服务名称作为key（如 `file-server`）
- 每个服务包含 `command` 和 `args` 字段
- `command` 为可执行命令
- `args` 为参数数组（可选）
