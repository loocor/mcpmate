# MCPMate 数据模型与业务逻辑

本文档描述了 MCPMate 的数据模型和业务逻辑，特别是关于服务器和工具的配置、启用和禁用机制。

## 数据库表结构

MCPMate 使用 SQLite 数据库存储配置信息，主要包含以下几组表：

1. **server_** 系列表：存储上游 MCP 服务器的基本配置信息
   - `server_config`：服务器基本信息（名称、类型、命令、URL等）
   - `server_args`：服务器命令行参数
   - `server_env`：服务器环境变量
   - `server_meta`：服务器元数据（描述、作者、网站等）

2. **config_suit_** 系列表：存储服务器和工具的启用状态配置
   - `config_suit`：配置套件基本信息（名称、类型、描述等）
   - `config_suit_server`：配置套件中的服务器及其启用状态
   - `config_suit_tool`：配置套件中的工具及其启用状态，包含UUID作为主键和可选的工具前缀名

## 业务逻辑

### 服务器配置与启用

1. `server_` 系列表用于记录上游 MCP 服务器运行所需的基本信息（命令、参数、环境变量等）。这些表内的服务器不存在启用与否的概念，它们只是配置信息的存储。

2. 在用户交互界面上，`server_` 系列表可按照场景等 `server_meta` 表中的字段维度进行分类展示，供用户查阅、开关其状态。

3. `server_` 系列表的数据不会直接作为 MCP 服务连接池的"任务表"。通过 `server_` 系列表由用户进行开关、成组（config_suit）后的数据进入 `config_suit_` 系列表中的记录，才是连接池决定初始化哪些服务、放行哪些工具的直接依据。

4. `config_suit_` 系列表有一套 default 数据集，允许用户增减其管理的服务和工具。这确保用户在没有创建新的 config_suit 时，也有一套基础的（样本配置）服务/工具可运行使用。

### 工具启用检查逻辑

工具启用检查逻辑在 `is_tool_enabled` 函数中实现，其逻辑如下：

1. 首先检查服务器在默认 config_suit 中是否启用。
2. 如果服务器在 suit 中被禁用，则工具也被禁用。
3. 如果服务器启用，则检查 suit 中是否有特定的工具配置，如果有则使用该配置的启用状态。
4. 如果没有找到任何配置，则默认工具是启用的。

### 服务器初始化逻辑

服务器初始化逻辑在 `load_server_config` 函数中实现，其逻辑如下：

1. 从数据库中获取所有在默认 config_suit 中启用的服务器。
2. 如果没有 default config_suit 或者 suit 中没有服务器配置，则加载所有服务器。
3. 否则，只加载在 default config_suit 中启用的服务器。
4. 为每个启用的服务器加载其参数、环境变量等配置信息。
5. 将配置信息转换为 MCPServerConfig 对象，并添加到配置中。

## 配置套件（Config Suit）

配置套件是一组服务器和工具的集合，用于管理哪些服务器和工具应该被启用。

## 工具前缀名（Prefixed Name）

为了避免多个服务中的同名工具冲突，MCPMate 支持为工具添加前缀名。前缀名存储在 `config_suit_tool` 表的 `prefixed_name` 字段中，用于在向下游客户端提供工具时区分不同服务的同名工具。

工具前缀名的格式通常为 `{server_name}:{tool_name}`，例如 `openai:chat` 和 `anthropic:chat`。这样，即使不同服务提供了同名的工具，客户端也可以通过前缀名来区分它们。

1. 每个配置套件有一个唯一的名称和类型（HostApp、Scenario、Shared）。
2. 配置套件可以包含多个服务器和工具，每个服务器和工具都有一个启用状态。
3. 默认情况下，系统会创建一个名为 "default" 的配置套件，类型为 Shared。
4. 当用户通过 API 启用或禁用服务器时，系统会更新 default 配置套件中的服务器启用状态。
5. 当系统启动时，只有在 default 配置套件中启用的服务器才会被初始化和连接。

## 未来计划

1. 配置套件可命名、分类、备注，只需要更新 config_suit 相应的字段确保可存储相应的数据。
2. 配置套件也可克隆复制，供用户在此基础上编辑调整出新的组合。
3. 下游 MCP client 只需一个兼容的 SSE / stdio / streamablehttp 通信模式的配置即可享用 MCPMate 中启用的 config_suit 的服务。
4. 在后期，如果能识别 client 身份标记，则可形成"专属"对应的 suit 服务/工具。
5. 对于 Cursor 等知名下游 MCP Client，其配置也来自于 config_suit，在 Client 视角配置页面中可直接选择现成的 config_suit，也可以自行选择所需的 server、tool。
6. 由于知名 MCP client 的配置文件路径可预判，因此它们可以享受近乎零配置的"福利"——一旦在 MCPMate 中配置好，MCPMate 会自动更新这些 client 的配置文件。
