# MCPMate Desktop 开发路线图

## 🎯 **当前开发重点**

### **Phase 1: 数据接口层重构 (当前阶段)**

**目标**：解决 macOS Desktop 应用的数据接口层问题，确保与后端 API 的完整集成。

## 📋 **核心技术问题清单**

### **问题 1: API 端点映射不完整**
**现状**：部分模块缺少对应的后端接口或映射不正确
**具体问题**：
- [ ] Tools 模块缺少独立的 API 端点，需要通过 Config Suits 获取
- [ ] Servers 的实例管理和健康检查接口未完全集成
- [ ] 部分模块的 CRUD 操作接口缺失
- [ ] Runtime 安装功能的 API 集成不完整

**解决方案**：
- [ ] 分析 openapi.json 中的完整端点列表
- [ ] 为每个模块建立完整的 API 端点映射表
- [ ] 实现缺失的 API 服务方法
- [ ] 验证所有端点的请求/响应格式

### **问题 2: 数据模型字段不匹配**
**现状**：Swift 模型与后端 API 响应结构存在字段差异
**具体问题**：
- [ ] 日期字段处理不一致（Date vs String）
- [ ] 嵌套对象的序列化/反序列化需要优化
- [ ] 可选字段的处理逻辑不统一
- [ ] 枚举类型的映射不完整

**解决方案**：
- [ ] 对比所有 Swift 模型与 API 响应的字段差异
- [ ] 统一日期字段的处理格式
- [ ] 完善 CodingKeys 映射
- [ ] 添加数据转换的单元测试

### **问题 3: 网络请求错误处理优化**
**现状**：网络请求错误处理较基础，缺少统一的状态管理
**具体问题**：
- [ ] 缺少统一的加载状态管理
- [ ] 错误信息展示不够用户友好
- [ ] 网络超时和重试机制不完善
- [ ] API 错误码的处理不统一

**解决方案**：
- [ ] 实现统一的网络请求状态管理
- [ ] 添加用户友好的错误信息展示
- [ ] 实现网络请求的重试机制
- [ ] 建立标准的错误处理流程

### **问题 4: 数据流验证不足**
**现状**：部分模块仍依赖 mock 数据，实时数据更新机制不完善
**具体问题**：
- [ ] 部分模块仍在使用 mock 数据
- [ ] API 响应数据到 UI 的转换链路需要验证
- [ ] 实时数据更新机制不完善
- [ ] 数据缓存策略不明确

**解决方案**：
- [ ] 移除所有 mock 数据依赖
- [ ] 验证完整的数据流转链路
- [ ] 实现实时数据更新机制
- [ ] 建立数据缓存和刷新策略
